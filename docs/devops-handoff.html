GymFlow MVP Deployment Handoff

DevOps Instructions for App + PostgreSQL Setup

This document explains how to deploy the GymFlow frontend, backend, and database for demo/UAT. The source code is already pushed to GitHub. The live database is not pushed to GitHub; DevOps must provision PostgreSQL and run the schema/seed steps listed here.

  • Repository github.com/tahamie/gymportal
  • Latest DB Handoff Commit 9b0e7af
  • Target Stage Controlled Demo / UAT

High-Level Architecture

GymFlow is a tenant-based product with separate Tenant Portal and Super Admin Portal flows.

Tenant Isolated
Frontend React + Vite static build served from dist/.
Backend Node.js HTTP API compiled from TypeScript.
Database PostgreSQL central DB plus isolated tenant DBs.

Environment Variables

Use real secrets on the server. Do not commit production passwords or tokens.

Required
Variable Purpose Example
PORT Backend API port. 4100
VITE_GYMFLOW_API_BASE_URL Frontend API base URL. Prefer reverse proxy path for same-domain deployment. /api
VITE_GYMFLOW_API_MODE Frontend API mode. backend
GYMFLOW_REPOSITORY Persistence mode. postgres
GYMFLOW_SEED_DEMO_DATA Whether seed script creates fake members/payments. Keep false for client demo entry testing. false
GYMFLOW_CENTRAL_DATABASE_URL Central Super Admin/platform DB. postgres://USER:PASSWORD@HOST:5432/gymflow_central
GYMFLOW_POSTGRES_ADMIN_URL Admin-capable connection for provisioning workflows. postgres://ADMIN:PASSWORD@HOST:5432/postgres
GYMFLOW_TENANT_DATABASE_URL_TEMPLATE Tenant DB connection template. Must include {database}. postgres://USER:PASSWORD@HOST:5432/{database}
A safe example file is included in the repo as .env.example. Production values must be stored in the deployment platform or server secret manager.

Database Setup

There is no live DB file to upload. DevOps must provision PostgreSQL and apply the schema.

PostgreSQL
  1. Create PostgreSQL server or managed PostgreSQL instance.
  2. Create central database, recommended name: gymflow_central.
  3. Create one tenant database per tenant, for demo: tenant_fitzone_khi.
  4. Apply backend/db/central.sql to the central database.
  5. Apply backend/db/tenant.sql to every tenant database.
  6. Set all GYMFLOW_* environment variables.
  7. For demo/UAT, run the seed script with GYMFLOW_SEED_DEMO_DATA=false.
npm run backend:seed-postgres

To remove fake members/payments from an existing seeded database:

npm run backend:clean-postgres
Production should not use demo passwords. The default seed is now clean and does not create fake members or payments.

Security Notes

Before public production.

  • Replace demo passwords.
  • Use HTTPS only.
  • Store secrets outside Git.
  • Enable backups and restore testing.
  • Configure CORS for real domains.

Production Hardening Still Required

The current build is suitable for controlled demo/UAT. These items are needed before real go-live.

Before Go-Live
  • Password hashing and real password verification
  • JWT/session expiry and refresh tokens
  • MFA-ready Super Admin login flow
  • Background workers for renewals and notifications
  • Tenant stats sync job
  • Provisioning job worker
  • WhatsApp/SMS/email provider credentials
  • Billing/payment provider production integration
  • Centralized logs and monitoring
  • Automated database backups
  • Role-based endpoint hardening
  • Production CORS and secret rotation
Recommended next step for DevOps: deploy this first as a password-protected UAT/demo environment, verify complete client flow, then harden for public production.
